# Architecture Decision: Proxy Routes vs Direct Frontend Access

**Date:** 2026-03-31
**Status:** Open question — build as-is now, revisit when payment service matures

## Question

Why do mesh clients need 20 proxy routes when Stripe/Chargebee let frontends
call them directly? Should our payment service behave the same way?

## Current State

The `create_billing_router()` generates 20 routes. Of those:

**7 essential (need secret key on backend):**
- Create checkout session, portal session, top-up session
- Checkout init (anti-fraud), checkout complete (verify)
- Webhook forwarding
- List plans (public pricing page)

**7 handled by Stripe Portal (zero proxy needed):**
- List/cancel subscriptions → Stripe Portal
- List invoices, invoice PDF → Stripe Portal
- List/set-default/remove payment methods → Stripe Portal

**4 billing-service-specific (not Stripe data):**
- Balance, usage summary, usage records, transactions
- These come from our billing service, not Stripe

## The Ideal Future

The payment service IS our "Stripe" — it should own the domain completely.
Frontends should call it directly (or use hosted pages) for everything.
Consumer backends only handle operations needing the secret key.

To get there, the payment service would need:
1. A scoped frontend token (like Stripe's `client_secret`) that consumers
   issue to their frontends for direct access
2. Or: accept the consumer's JWT by validating via auth service callback
3. Or: serve its own hosted dashboard (like Stripe Dashboard)

## Decision

**Build as-is now.** The 20-route proxy works, is tested (53 UJ tests GREEN),
and doesn't hurt performance. The extra routes (subscriptions, invoices,
methods) give consumers flexibility to build custom UIs if Stripe Portal
isn't sufficient.

**Revisit when:**
- A second mesh service integrates billing (proves the pattern or exposes friction)
- Payment service adds a frontend token / session token for direct access
- Someone builds a hosted billing dashboard in the payment service

The proxy routes don't create technical debt — they're generated by the library,
not hand-written. Removing them later is a config change, not a rewrite.
